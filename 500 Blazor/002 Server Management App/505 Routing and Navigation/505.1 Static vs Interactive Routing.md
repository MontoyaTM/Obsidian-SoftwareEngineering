# Static Routing

In Blazor, **Static routing** means the server renders a component on request, streams the HTML to the browser, and stops — there's no persistent connection, no JS interop after the initial load, and no client-side DOM diffing.

- Navigation to a new route fires a full HTTP request cycle.
- It's essentially server-side rendering (SSR).

# Interactive Routing

**Interactive routing** keeps a live channel open — either a SignalR WebSocket (Blazor Server) or a .NET runtime in the browser (Blazor WebAssembly). The router intercepts navigation events on the client, loads the new component without a full page reload, and the component lifecycle (state, events, re-renders) all happen over that channel.

- You get SPA-style behavior without writing JavaScript.

The key nuance is that Blazor's `InteractiveRouteContext` can be applied globally (in `App.razor` via `@rendermode`) or per-component — so you can mix static pages with interactive islands on the same app.


![[505.1 Static vs Interactive Routing-1775832522496.webp|825]]

**Render modes that matter for routing:**

- <span style="color:rgb(129, 253, 131)">@rendermode InteractiveServer</span> — keeps the component on the server, tunnels UI updates over SignalR. Low client footprint, but latency-sensitive.
- <span style="color:rgb(129, 253, 131)">@rendermode InteractiveWebAssembly</span> — ships a .NET runtime to the browser; fully client-side after first load.
- <span style="color:rgb(129, 253, 131)">@rendermode InteractiveAuto</span> — starts as Server for fast first render, then silently transitions to WASM once the runtime is cached.
- No @rendermode (default) — pure static SSR.
